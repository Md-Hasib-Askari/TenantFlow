Accumulated wisdom,

Every architect knows standard structural solutions — how to frame a doorway, how to lay a foundation — without inventing them fresh each time. Design patterns are the same idea for software: proven, reusable solutions to recurring design problems.

The concept was popularized by the "Gang of Four" — Gamma, Helm, Johnson, and Vlissides — in their 1994 book Design Patterns: Elements of Reusable Object-Oriented Software, which catalogued 23 classic patterns.

// 3.1 Singleton Pattern

One instance. One global
access point.

Think of the CEO of a company — there is exactly one, and everyone refers to the same person. A Singleton guarantees a class has only one instance for the lifetime of the application.

Without it, a DatabaseConnectionPool instantiated freely wastes resources, creates conflicts, and breaks consistency across the app.

Creational — Singleton 08 / 41

// 3.1 Singleton Pattern

Thread-safe: double-checked locking

// Two null-checks avoid lock overhead once initialized
public class ConfigurationManager
{
  private static volatile ConfigurationManager _instance;
  private static readonly object _lock = new object();

  private ConfigurationManager()
  {
    // Load settings once, on first access
  }

  public static ConfigurationManager Instance
  {
    get
    {
      if (_instance == null)
        lock (_lock)
          if (_instance == null)
            _instance = new ConfigurationManager();
      return _instance;
    }
  }
}

// 3.1 Singleton Pattern

The modern C# way: Lazy<T>

public class AppLogger
{
  private static readonly Lazy<AppLogger> _lazy =
    new Lazy<AppLogger>(() => new AppLogger());

  private readonly List<string> _logBuffer = new();

  private AppLogger() { }

  public static AppLogger Instance => _lazy.Value;

  public void Log(string message)
  {
    var entry = $"[{DateTime.Now:HH:mm:ss}] {message}";
    _logBuffer.Add(entry);
    Console.WriteLine(entry);
  }
}

// 3.1 Singleton Pattern

The anti-pattern: mutable shared state

// DANGER — shared across ALL users in ASP.NET Core
public class UserSession
{
  public string CurrentUserId { get; set; }
}
// One user's request WILL overwrite another's field

Use When

  • Resource is expensive to create
  • Single coordination point needed
  • State is stateless or thread-safe

Avoid When

  • Object holds mutable per-user state
  • Unit testing is critical to the team
  • Concurrent requests could race a field

// 3.2 Factory Method Pattern

Let subclasses decide
what to build

A logistics company's createTransport() makes trucks by default; a sea-freight subclass overrides it to make ships. Core business logic never cares how transport was created.

// BAD — client tightly coupled to concrete types
public class NotificationService
{
  public void Send(string type, string message)
  {
    INotification notification;
    if (type == "email")
      notification = new EmailNotification();
    else if (type == "sms")
      notification = new SmsNotification();
    else
      throw new ArgumentException($"Unknown: {type}");
  }
  // Every new type forces a code change — violates OCP
}

// 3.2 Factory Method Pattern

The Factory Method in practice

public interface INotification
{
  void Send(string to, string message);
}

public class EmailNotification : INotification
{
  public void Send(string to, string msg) =>
    Console.WriteLine($"[EMAIL] {to}: {msg}");
}

public abstract class NotificationFactory
{
  public abstract INotification CreateNotification();

  public void Notify(string to, string msg) =>
    CreateNotification().Send(to, msg);
}

public class EmailNotificationFactory : NotificationFactory
{
  public override INotification CreateNotification() =>
    new EmailNotification();
}
Creational — Factory Method 13 / 41

// 3.2 Factory Method Pattern

Registry-based — the practical shape

public class PaymentProcessorFactory : IPaymentProcessorFactory
{
  private readonly Dictionary<string, Func<IPaymentProcessor>> _registry;

  public PaymentProcessorFactory(IServiceProvider sp)
  {
    _registry = new(StringComparer.OrdinalIgnoreCase)
    {
      ["stripe"] = () => sp.GetRequiredService<StripePaymentProcessor>(),
      ["paypal"] = () => sp.GetRequiredService<PayPalPaymentProcessor>(),
    };
  }

  public IPaymentProcessor Create(string provider) =>
    _registry.TryGetValue(provider, out var f)
      ? f()
      : throw new NotSupportedException(provider);
}

// 3.4 Builder Pattern

A fluent, self-documenting API

public interface IEmailMessageBuilder
{
  IEmailMessageBuilder From(string email);
  IEmailMessageBuilder To(string email);
  IEmailMessageBuilder WithSubject(string subject);
  IEmailMessageBuilder WithBody(string body, bool isHtml = false);
  EmailMessage Build();
}

public class EmailMessageBuilder : IEmailMessageBuilder
{
  private readonly EmailMessage _email = new();

  public IEmailMessageBuilder From(string email)
  {
    _email.From = email;
    return this;
  }
  // To(), WithSubject(), WithBody() follow the same shape
}

// 3.4 Builder Pattern

Dynamic SQL query builder

public class SqlQueryBuilder
{
  private string _table;
  private readonly List<string> _where = new();

  public SqlQueryBuilder From(string table)
  {
    _table = table;
    return this;
  }

  public SqlQueryBuilder Where(string col, object val)
  {
    _where.Add($"{col} = @{col}");
    return this;
  }

  public string Build()
  {
    var sql = $"SELECT * FROM {_table}";
    if (_where.Any())
      sql += $" WHERE {string.Join(" AND ", _where)}";
    return sql;
  }
}

// Usage
var sql = new SqlQueryBuilder()
  .From("Users").Where("IsActive", true).Where("Age", 18)
  .Build();

// 3.5 Prototype Pattern

Clone the template,
not the factory line

Like cell mitosis — a cell clones itself into a nearly identical copy. Used when building an object from scratch is expensive.

public class Employee
{
  public string Name { get; set; }
  public Address Address { get; set; }

  // SHALLOW: reference fields still point to the same object
  public Employee ShallowClone() =>
    (Employee)this.MemberwiseClone();

  // DEEP: a fully independent copy
  public Employee DeepClone() => new Employee
  {
    Name = this.Name,
    Address = new Address { City = this.Address.City }
  };
}
> shallow.Address.City = "New York";
> original.Address.City  →  "New York"   // shallow bug!

// 3.5 Prototype Pattern

Prototype registry

public interface IPrototype<T>
{
  T Clone();
}

public class DocumentTemplate : IPrototype<DocumentTemplate>
{
  public string Title { get; set; }
  public List<string> Sections { get; set; }

  public DocumentTemplate Clone() => new DocumentTemplate
  {
    Title = this.Title,
    Sections = new List<string>(this.Sections)
  };
}

public class DocumentTemplateRegistry
{
  private readonly Dictionary<string, DocumentTemplate> _templates = new();

  public DocumentTemplate Get(string name) =>
    _templates[name].Clone(); // Always return a CLONE
}

// 4.1 Strategy Pattern

Interchangeable algorithms

Google Maps: you choose the strategy — fastest, shortest, avoid highways. The app itself never changes, only the routing algorithm does.

public interface IDiscountStrategy
{
  decimal CalculateDiscount(Order order);
  string Description { get; }
}

public class FirstOrderDiscountStrategy : IDiscountStrategy
{
  public string Description => "First Order: 15% Off";

  public decimal CalculateDiscount(Order order) =>
    order.IsFirstOrder
      ? Math.Round(order.TotalAmount * 0.15m, 2)
      : 0m;
}
Behavioral — Strategy 26 / 41

// 4.1 Strategy Pattern

Composite strategy, swapped at runtime

public class CompositeDiscountStrategy : IDiscountStrategy
{
  private readonly List<IDiscountStrategy> _strategies;

  public decimal CalculateDiscount(Order order)
  {
    var total = _strategies.Sum(s => s.CalculateDiscount(order));
    var cap = order.TotalAmount * 0.40m;
    return Math.Min(total, cap);
  }
}

// Context — swaps strategy at runtime
public class OrderProcessor
{
  private IDiscountStrategy _strategy = new NoDiscountStrategy();

  public void SetStrategy(IDiscountStrategy s) => _strategy = s;

  public OrderSummary Process(Order order)
  {
    var discount = _strategy.CalculateDiscount(order);
    return new OrderSummary { DiscountAmount = discount };
  }
}

// 4.2 Observer Pattern

One change, many reactions

A YouTube channel does not know its subscribers individually — it broadcasts, and every subscriber reacts on its own.

public class OrderService : ISubject<OrderStatusChangedEvent>
{
  private readonly List<IObserver<OrderStatusChangedEvent>> _observers = new();

  public void Subscribe(IObserver<OrderStatusChangedEvent> o) =>
    _observers.Add(o);

  public void Notify(OrderStatusChangedEvent data)
  {
    foreach (var observer in _observers.ToList())
      observer.OnNext(data);
  }

  public void UpdateOrderStatus(int orderId, string status) =>
    Notify(new OrderStatusChangedEvent
    {
      OrderId = orderId, NewStatus = status
    });
}

// 4.3 Command Pattern

Requests as objects — undo included

A restaurant order slip: the waiter does not cook — they hand the kitchen a command that can be replicated or cancelled.

public class CommandHistory
{
  private readonly Stack<ICommand> _undoStack = new();

  public void Execute(ICommand cmd)
  {
    cmd.Execute();
    _undoStack.Push(cmd);
  }

  public void Undo()
  {
    var cmd = _undoStack.Pop();
    cmd.Undo();
  }
}
> history.Execute(Insert "Hello", 0)
> history.Execute(Insert " World", 5)  →  "Hello World"
> history.Undo()                       →  "Hello"

// 4.4 Template Method Pattern

Fixed skeleton, swappable steps

Making a burger: toast bun, add sauce, add patty, wrap. The sequence never changes — a veggie subclass overrides only "add patty."

public abstract class ReportGenerator
{
  // Template method — sealed, order cannot change
  public sealed string GenerateReport(ReportParameters p)
  {
    var data = FetchData(p);
    var valid = ValidateData(data);
    var formatted = FormatData(valid);
    return RenderReport(formatted);
  }

  protected abstract object[] FetchData(ReportParameters p);
  protected abstract object[] FormatData(object[] data);
  protected abstract string RenderReport(object[] data);
}

// 4.5 Chain of Responsibility

Pass it along the chain

Customer support escalation: chatbot → junior agent → senior agent → management. Each handler resolves the ticket or passes it on.

public abstract class SupportHandler
{
  protected SupportHandler _next;

  public SupportHandler SetNext(SupportHandler h)
  {
    _next = h;
    return h;
  }

  protected string PassToNext(SupportTicket t) =>
    _next?.Handle(t) ?? $"[UNRESOLVED] Ticket #{t.Id}";
}
// chatbot.SetNext(junior).SetNext(senior).SetNext(mgmt);

// 4.5 Chain of Responsibility

ASP.NET middleware IS the chain

public class RequestLoggingMiddleware
{
  private readonly RequestDelegate _next;

  public async Task InvokeAsync(HttpContext context)
  {
    _logger.LogInformation("{Method} {Path}",
      context.Request.Method, context.Request.Path);

    await _next(context); // Pass to the next handler

    _logger.LogInformation("Completed {Code}",
      context.Response.StatusCode);
  }
}

// 4.6 Iterator Pattern

Traverse without exposing internals

Built into C# as IEnumerable<T> / IEnumerator<T> — every foreach loop is this pattern in action.

public class Playlist : IEnumerable<Song>
{
  private readonly List<Song> _songs = new();

  public IEnumerator<Song> GetEnumerator() =>
    _songs.GetEnumerator();
}

// Lazy tree traversal with yield return
private IEnumerable<T> DepthFirst(TreeNode<T> node)
{
  yield return node.Value;
  foreach (var child in node.Children)
    foreach (var val in DepthFirst(child))
      yield return val;
}

// Usage — works with foreach AND LINQ
var longSongs = playlist.Where(s => s.Duration > TimeSpan.FromMinutes(5));

// 4.8 Mediator Pattern

Talk through the tower,
not to each other

Air traffic control: planes never talk to each other — every message routes through the tower, reducing coupling.

public class ChatRoom : IChatMediator
{
  private readonly List<ChatUser> _users = new();

  public void SendMessage(string message, ChatUser sender)
  {
    foreach (var user in _users.Where(u => u != sender))
      user.Receive(message, sender);
  }
}

// alice.Send("Hello everyone!"); → routed through ChatRoom

// Decision Guide

Which pattern do I reach for?

Controlling Creation

  • One instance only → Singleton
  • One product, delegated → Factory Method
  • Family of related products → Abstract Factory
  • Complex step-by-step build → Builder
  • Clone existing objects → Prototype

Controlling Behavior

  • Swap algorithms at runtime → Strategy
  • Notify many on change → Observer
  • Encapsulate a request → Command
  • Fixed skeleton, override steps → Template Method
  • Pass a request through handlers → Chain of Resp.
  • Iterate without exposing internals → Iterator
  • Behavior changes with state → State
  • Reduce coupling via coordinator → Mediator

// Key Takeaways

What to carry into your next PR

  • Creational patterns answer "how is this made" — Behavioral patterns answer "how do objects talk."
  • ASP.NET Core is already built from these: Builder, DI singletons, middleware chains, MediatR.
  • Prefer DI-managed lifetimes over hand-rolled singletons — testability wins.
  • Strategy + Factory is the standard shape for swappable business rules like pricing or payments.
  • Deep-clone anything with reference-type fields, or Prototype will bite you.
  • Patterns are shared vocabulary, not rules — reach for them where they remove real duplication.
